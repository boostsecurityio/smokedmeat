# Audit - 2026-04-17

## Scope

Read-only audit of production code across `cmd/` and `internal/`, focused on exploit generation, deployment, support signaling, callback staging, cache poisoning, persistence, and operator UX flows that can misrepresent runtime capability.

## Audit Log

### Confirmed findings

1. `internal/lotp/payload.go` - callback-driven LOTP deployment silently falls back to `id` for several "supported" families.
   - `GenerateFiles()` passes only `CallbackURL` into `RecommendBestPayload()` for non-script LOTP families.
   - `pip`, `yarn`, `cargo`, and `make` ignore `CallbackURL` in their primary generated payloads and default to `id` when `Command` is empty.
   - Deployment materializes only the first payload, so `pip` never reaches its later `requirements.txt` callback variant in the real deploy path.
   - Relevant refs:
     - `internal/lotp/payload.go:45`
     - `internal/lotp/payload.go:50`
     - `internal/lotp/payload.go:156`
     - `internal/lotp/payload.go:251`
     - `internal/lotp/payload.go:291`
     - `internal/lotp/payload.go:331`
     - `internal/lotp/payload.go:420`
     - `internal/lotp/payload.go:467`

2. LOTP delivery is advertised far more broadly than the backend can actually deploy.
   - Any `untrusted_checkout_exec` finding with `lotp_tool` or `lotp_action` metadata is offered as `DeliveryLOTP` in the TUI.
   - Preflight marks LOTP availability based only on repo and PR capability, not on generator support for the specific LOTP family.
   - Backend deployment then passes either `LOTPTool` or `LOTPAction` into `lotp.GenerateFiles()`, which only supports script tools plus a narrow subset of package/tool families.
   - Result: the UI can present LOTP as deployable for unsupported action-backed or catalog-only variants that will fail server-side with `unsupported LOTP tool`.
   - Relevant refs:
     - `internal/counter/tui/phase.go:247`
     - `internal/pantry/assets.go:175`
     - `internal/kitchen/github_preflight.go:642`
     - `internal/counter/tui/deploy.go:122`
     - `internal/kitchen/github.go:1608`
     - `internal/kitchen/github.go:1152`
     - `internal/kitchen/github.go:1154`
     - `internal/lotp/payload.go:45`
     - `internal/lotp/payload.go:384`

3. Non-script LOTP deployment ignores `lotp_targets` and always writes fixed root filenames.
   - The TUI surfaces `LOTPTargets` to the operator as the intended file set.
   - Deployment forwards those targets into `lotp.GenerateFiles()`.
   - Only `bash`, `powershell`, and `python` honor the supplied targets. `npm`, `pip`, `yarn`, `cargo`, and `make` ignore them and emit fixed paths such as `package.json`, `setup.py`, `.yarnrc.yml`, `build.rs`, and `Makefile`.
   - Result: path-gated or subdirectory LOTP findings can look correctly targeted in the UI but produce PRs that modify the wrong files.
   - Relevant refs:
     - `internal/counter/tui/layout.go:2208`
     - `internal/counter/tui/layout.go:2244`
     - `internal/counter/tui/deploy.go:136`
     - `internal/kitchen/github.go:1152`
     - `internal/lotp/payload.go:45`
     - `internal/lotp/payload.go:306`
     - `internal/lotp/payload.go:467`

4. Brisket LOTP detection drops relative paths, so path-aware catalog entries are missed.
   - `lotpDetect()` walks the repo but keeps only `filepath.Base(rel)` for each file.
   - `FindByFile()` and `matchFile()` only match exact filenames or a limited suffix glob.
   - Techniques whose signatures depend on subpaths or directory names, such as `.bundle/config`, `.cargo/config.toml`, `.husky/pre-commit`, and `.mvn/`, cannot be detected reliably from the Brisket LOTP helper even though they exist in the catalog.
   - Relevant refs:
     - `internal/brisket/inject.go:156`
     - `internal/brisket/inject.go:165`
     - `internal/lotp/lotp.go:77`
     - `internal/lotp/lotp.go:102`
     - `internal/lotp/lotp.go:236`
     - `internal/lotp/lotp.go:308`

5. Secret reference extraction only supports dot notation and misses bracket notation.
   - Workflow metadata extraction scans for `secrets.NAME` only.
   - App action secret extraction also only recognizes `secrets.NAME`.
   - GitHub Actions expressions written as `secrets['NAME']` or `secrets["NAME"]` will be skipped, which means secret inventory, app secret typing, and downstream pivot suggestions can be incomplete even when the workflow clearly references the secret.
   - The workflow-level secret scan is also substring-based, so strings that merely contain `secrets.` without being a real expression can be over-counted.
   - Relevant refs:
     - `internal/poutine/analyzer.go:516`
     - `internal/poutine/analyzer.go:651`
     - `internal/poutine/analyzer.go:919`
     - `internal/poutine/analyzer.go:932`
     - `internal/poutine/analyzer_test.go:755`
     - `internal/poutine/analyzer_test.go:976`

6. Kitchen stager metadata is mutated outside the store lock after `Get()` returns an internal pointer.
   - `StagerStore.Get()` returns the live `*RegisteredStager` from the map without cloning.
   - GitHub deploy handlers then mutate `stager.Metadata[...]` directly after `Get()`, outside any store lock.
   - Callback handling and persistence paths also read and serialize the same `Metadata` map.
   - Result: concurrent callback traffic during deploy completion can race on shared stager state, including a possible `concurrent map read and map write` panic on the metadata map.
   - Relevant refs:
     - `internal/kitchen/stager.go:110`
     - `internal/kitchen/github.go:1491`
     - `internal/kitchen/github.go:1531`
     - `internal/kitchen/github.go:1572`
     - `internal/kitchen/github.go:1626`
     - `internal/kitchen/callbacks.go:117`
     - `internal/kitchen/handlers.go:919`
     - `internal/kitchen/stager.go:387`
